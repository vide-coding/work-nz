## Context

当前系统使用固定的五种目录类型（`code`, `docs`, `ui_design`, `project_planning`, `custom`），每种类型与特定功能强绑定。这种设计限制了用户根据实际需求灵活组织项目资源的能力。

本次重构将引入**模块化目录系统**，核心变化：

- 目录不再由固定类型定义，而是通过启用**模块**获得功能
- 每个模块通过 JSON Schema 描述其配置和能力
- **模块互斥**：一个目录只能启用一个模块
- 支持**目录模板**在项目创建时快速初始化目录结构

## Goals / Non-Goals

**Goals:**

1. 实现模块化的目录系统，支持模块的定义、注册、配置和启用
2. 实现目录模板系统，支持模板的创建、应用、导出和共享
3. 将 `code` 功能保留为内置 `git` 模块
4. 实现模块的 JSON Schema 定义和验证机制
5. 实现项目创建时的模板选择流程

**Non-Goals:**

1. 不支持一个目录同时启用多个模块（保持互斥）
2. 不支持模块的动态加载（模块需预注册到系统）
3. 不支持模块间的依赖关系（模块独立运行）
4. 不实现远程模板市场（仅支持本地和导出/导入）

## Decisions

### 1. 模块互斥 vs 模块组合

**决策**：采用**模块互斥**（一个目录只能启用一个模块），而非模块组合。

**理由**：

- 保持概念简单性：每个目录有明确单一的职责
- 避免配置冲突：不同模块可能有重叠的配置项
- UI 清晰性：用户知道当前目录的确切功能

**替代方案**：模块组合（一个目录可同时启用多个模块）被否决，因为会增加复杂度和潜在的模块间冲突。

### 2. 模块配置使用 JSON Schema

**决策**：模块的配置结构通过 **JSON Schema** 定义，配置数据以 JSON 格式存储。

**理由**：

- 标准化：JSON Schema 是业界标准，有丰富的工具支持
- 动态表单：可以根据 Schema 自动生成配置界面
- 验证：可以自动验证配置数据的合法性
- 类型安全：可以生成 TypeScript 类型定义

**替代方案**：使用 TypeScript 接口或自定义 DSL 被否决，因为需要额外的解析和验证实现。

### 3. 模板数据存储在本地文件系统

**决策**：目录模板以 JSON 文件形式存储在本地文件系统中。

**理由**：

- 简单性：无需数据库，文件操作直观
- 可移植性：模板文件可以轻松导出和导入
- 版本友好：可以使用 Git 管理模板文件
- 用户控制：用户可以直接编辑模板文件

**替代方案**：使用数据库存储或使用云存储被否决，因为会增加复杂度和外部依赖。

### 4. `code` 功能保留为 `git` 模块

**决策**：保留原有的 `code` 功能，但将其封装为内置的 `git` 模块。

**理由**：

- 向后兼容：现有项目的 `code` 目录可以继续正常工作
- 功能保留：Git 仓库管理是系统的核心功能之一
- 模块平等：`git` 模块与其他模块地位平等，遵循相同的配置机制

**迁移策略**：现有 `code` 类型的目录在数据迁移时自动关联到 `git` 模块。

## Risks / Trade-offs

### 1. 向后兼容性风险

**风险**：数据模型的大幅变更可能导致现有项目数据丢失或损坏。

**缓解措施**：

- 实现完整的数据迁移脚本，将旧的 `DirectoryType` 映射到新的 `Module`
- 在迁移前创建数据备份
- 提供回滚机制
- 添加版本检测，阻止新旧版本混合使用

### 2. 性能风险

**风险**：JSON Schema 验证可能影响性能，特别是在大量配置验证时。

**缓解措施**：

- 缓存解析后的 Schema，避免重复解析
- 只在配置变更时验证，不重复验证静态配置
- 使用轻量级的 JSON Schema 验证库
- 考虑配置变更的防抖处理

### 3. 用户体验风险

**风险**：模块概念增加了认知负担，用户可能感到困惑。

**缓解措施**：

- 提供清晰的默认配置，降低配置复杂度
- 提供模板系统，让用户可以直接使用预设配置
- 在 UI 中提供清晰的模块说明和示例
- 提供配置预览功能，让用户在保存前看到效果

### 4. 扩展性风险

**风险**：模块系统设计不当可能导致后期扩展困难。

**缓解措施**：

- 预留扩展点，如模块钩子（hooks）系统
- 保持模块接口的简洁性，避免过度设计
- 使用版本化的模块 API，允许模块独立演进
- 考虑模块的命名空间，避免冲突

## Migration Plan

### 数据迁移

1. **备份阶段**
   - 迁移前自动创建数据备份
   - 记录迁移前的版本号

2. **结构迁移**
   - 创建新的 `modules` 表/配置
   - 创建新的 `directories` 表/配置
   - 创建 `directory_templates` 表/配置

3. **数据映射**

   ```
   旧数据                      新数据
   ─────────────────────────────────────────
   DirectoryType('code')  →   Module('git')
   DirectoryType('docs')  →   Module('file') + 模板
   DirectoryType('ui_design') → Module('file') + 模板
   DirectoryType('project_planning') → Module('task') + 模板
   DirectoryType('custom') → Module('file')
   ```

4. **验证阶段**
   - 验证迁移后的数据完整性
   - 对比迁移前后的关键数据
   - 运行基本功能测试

5. **清理阶段**
   - 删除旧的 `DirectoryType` 数据
   - 更新应用版本号
   - 记录迁移日志

### 回滚策略

如果在迁移后发现问题，可以执行回滚：

1. 使用迁移前创建的备份恢复数据
2. 恢复到旧版本的应用代码
3. 清理新创建的数据结构（可选，保留作为历史记录）

### 部署计划

1. **阶段一：代码准备**
   - 在 feature 分支完成所有开发
   - 代码审查和测试
   - 准备迁移脚本

2. **阶段二：灰度测试**
   - 在测试环境执行完整迁移
   - 验证所有功能正常
   - 性能测试

3. **阶段三：正式发布**
   - 发布新版本
   - 自动执行数据迁移
   - 监控错误和性能

4. **阶段四：后续优化**
   - 收集用户反馈
   - 优化迁移体验
   - 完善文档

## Open Questions

1. **模块的版本管理**：模块的 JSON Schema 是否需要有版本号？如何实现模块的平滑升级？

2. **模板的版本控制**：模板导出/导入时是否需要包含版本信息？如何处理模板格式的不兼容变更？

3. **模块间的数据共享**：虽然目录只能启用一个模块，但不同目录的数据是否需要在某种程度上的关联？例如，任务模块的目录是否可以引用 Git 模块的提交？

4. **权限和访问控制**：在多用户场景下，模板和模块的权限如何管理？

5. **性能优化**：如果用户创建了大量目录，每个目录都有配置数据，如何优化数据加载和验证性能？

6. **国际化**：模块的显示名称、描述、配置项的标签等是否需要国际化支持？如何实现？

7. **调试和排错**：当模块配置出错时，如何提供清晰的错误信息和调试支持？
