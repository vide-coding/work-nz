## 新增需求

### 需求：使用 JSON Schema 定义模块

每个模块应由描述其配置结构和能力的 JSON Schema 定义。

#### 场景：模块 schema 验证

- **当** 模块向系统注册
- **那么** 系统应验证模块的 JSON Schema
- **并且** 仅当 schema 有效时模块才可用

### 需求：模块配置存储

系统应将模块配置作为 JSON 数据存储，并根据模块的 schema 进行验证。

#### 场景：目录模块配置

- **当** 用户创建带模块的目录
- **那么** 系统应存储模块特定配置
- **并且** 配置应根据模块的 JSON Schema 进行验证

### 需求：模块能力声明

每个模块应将其能力声明为能力字符串列表。

#### 场景：基于能力的 UI 渲染

- **当** 系统渲染带模块的目录
- **那么** UI 应仅显示与模块声明能力对应的操作
- **并且** 不支持的操作应被隐藏或禁用

### 需求：互斥模块强制

系统应强制目录一次只能启用一个模块。

#### 场景：模块切换

- **当** 用户尝试在已具有模块的目录上启用模块
- **那么** 系统应要求用户先禁用现有模块
- **并且** 系统应警告如果配置将丢失时的数据丢失风险

### 需求：模块注册系统

系统应为模块提供注册机制。

#### 场景：内置模块注册

- **当** 系统初始化
- **那么** 所有内置模块（git、task、file）应自动注册
- **并且** 其 JSON Schema 应被加载和验证

### 需求：模块配置 UI 生成

系统应根据模块的 JSON Schema 动态生成配置 UI 表单。

#### 场景：模块配置表单

- **当** 用户配置启用了模块的目录
- **那么** 系统应根据模块的 JSON Schema 渲染表单
- **并且** 表单字段应包括 schema 中定义的标签、类型和验证
